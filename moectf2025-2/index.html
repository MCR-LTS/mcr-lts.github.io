Speed —— 暴力 Patch

也是在上篇中被我暂时舍掉的一种方法：

找到反汇编最后关闭窗口的函数层（如 ExitProcess 或某处的 return），用 Keypatch 改成 jmp $，程序弹出来之后就会直接卡死。

如要修改，理论上最完美的位置是修改后两行。依旧在选中伪代码后，跳转至汇编页面。
然而 Keypatch 不认 jmp $，那直接复制本条指令的地址再 jmp addr 就行。
Edit -> Patch program -> Apply patches to，保存为一个新的 exe，直接打开就能看到被冻结的 flag。

不过本方法有一定的风险，CPU 单核占用率会瞬间飙升，程序必须强制退出，而且一旦不小心拖动其他窗口挡住了它，被挡住的部分就会永远消失。
或许更保险的方法是把 sleep(1u) 改成 sleep(114514u)。

Catch —— 动态步过

它能在运行时正常解密并验证，说明 CPU 根本不在乎 IDA 是不是看懂了。用 x64dbg 跑起来，在异常发生的触发点下断点，然后疯狂单步步过。跟着程序的执行流，你会眼睁睁看着它走进那个被隐藏的 catch 块，这时候寄存器里通常会暴露 ROT13 的过程甚至最终比对的结果。
静态分析可能会用尽各种花招骗你，但 CPU 跑过的指令永远不会撒谎。

严格来讲，这不算一种解法，用 IDA 都快要解决了，突然把 x64dbg 拉进来，某种程度上是脱裤子放那个。

聊聊这个脚本与 Z3 脚本的区别吧。

Python 爆破操作的是内存中的具体数值。char_code 是具体的整数，b_val = …… 这一行执行完毕后，b_val 内存里存放的就是一个计算好的具体数字。
Z3 脚本构建的是抽象语法树。flag[i] = BitVec(...) 是一个符号，执行 b[i] = …… 时，它并没有做任何实际的运算。b[i] 实际上是一个 BitVecRef 对象，记录了等式右侧的动作。

Python 爆破通过观察发现 b[i-1] 必须等于 a[i-1]，这直接斩断了前后字符的依赖链。每个字符的计算被降维成了完全独立的方程，一个单层嵌套循环就能解开。
Z3 脚本保留了原始的深层嵌套依赖。由于 b[i-1] 是一个巨大的符号表达式，b[i] 的表达式树将变得越来越庞大。计算 b[33] 时，它的内部包含了从 b[0] 到 b[32] 所有符号的嵌套异或逻辑，Z3 必须在全局范围内同时处理这 34 长串的表达式。

A cup of tea —— 用 C 语言

你吐槽用 Python 被各种大数溢出、& 0xFFFFFFFF 恶心坏了。 其实，CTF 界逆向 TEA/AES 等标准算法，最好的方法是用 C 语言手搓！

直接把 C 加密算法复制下来，把 += 批量替换成 -=，把步骤倒过来，稍微改改。unsigned int 是 32 位的，自带溢出截断，根本不用处理位运算越界问题。

也没啥好解析的，撑死让 AI 再给我讲讲 unsigned int。

顾名思义，没有符号，范围比 int 翻一番。一个用 31 位存数值，另一个 32 位全都用来存。

其优点包括：

天然的溢出截断：密码学算法经常会让数值爆炸，当数值爆表时，unsigned int 会像汽车里程表一样，自动清零从头开始算，完美等价于 Python 里的高精度截断。

纯粹的位运算：在进行 >> 时，普通的负数 int 右移会在高位补 1（为了保持负号），这会彻底破坏加密逻辑。而 unsigned int 右移时高位永远补 0（逻辑右移），保证了底层二进制比特流的绝对纯净。

无视补码假象：负数存进它时，底层的 32 位二进制毫无改变，只是 C 语言把它当成了大正数来处理，这对纯粹的数据解密来说非常方便。

对于那段让我头皮发麻的 *(unsigned int *)(&result[j * 8]) = v24，可这样解读：
&result[j * 8]：把第 j * 8 号抽屉的门牌号记了下来，在编译器眼里，这是一张“指向单人抽屉的门牌号”（底层类型是 char *，长度 1 字节）。
(unsigned int *)!!：强行把记录改写成“这是一个大抽屉的入口”，让编译器把这个地址当成了一个能装 4 字节（unsigned int）的大空间起点。

双击 ZF 即可反转值，继续运行……好吧依然是失败的，第七难的题怎么能用这么低级的方法。

其他

Catch —— 换用 Ghidra

太大了我电脑装不下，过。

ezpy —— 使用 dis

遇到很多新版本 Python 的 pyc，pycdc 会崩溃。可以写一个简短的 import dis; import marshal; main_code = marshal.loads(...) ; dis.dis(main_code)，直接看底层 Python 指令推导。

感觉适用性不强，就不验证了，我想不到有哪些题需要这么做。

flower —— IDAPython 自动化除花

既然 jz/jnz 这套花指令在题里长得都一样，完全可以让 ChatGPT 写一段脚本：“在整个内存段里搜索这串固定字节，把它们替换为 NOP” 。一键执行，IDA 瞬间清清爽爽。

这明摆着让你问 AI，我就不验证了。

ezandroid —— 解压 grep

APK 文件的本质是一个压缩包。大多数签到题甚至不需要装 JADX 环境，把后缀改成 .zip 解压，拿出里面的 classes.dex。strings classes.dex | grep -i "moectf{"，如果有硬编码明文，瞬间就出来了。

本题为 base64 编码，因此该方法不适用。但是对签到题来讲是个非常好的思路。
或者向某位直觉大神看齐直接拿着 “moectf{” 去 base64 编码再搜索

ezandroid —— MT 管理器

点击 Dex 编辑器，全局搜索 base64 或中文字符。

have_fun —— 扒光底裤

碰到 Win32 窗口题，下个 Resource Hacker， 把 exe 丢进去，里面有个 Dialog 文件夹，点开后能以图形化界面看到题目的所有弹窗、隐藏按钮、文本框 ID 甚至是假 Flag 字符，还能直接修改重新打包！很多信息看资源表就能拿到。

我们 exe 有自己的 AssetStudio。可惜的是这道题找不到啥隐藏信息。

在快乐的 AI 问问问之后，就要处理棘手的最后六道题了。
