栈是什么

一种后进先出的数据结构，满足调用函数的方式（调用时父前子后，返回时子先父后）。
栈支持 push 压入与 pop 弹出并存储到指定寄存器/内存中，生长方向从高地址到低地址。
pop 后，栈中的数据并没有被清除，只是无法直接访问。

栈帧是什么

一种专门用于保存函数调用过程中的参数、返回地址、本地变量等信息的栈。
ebp 指向栈底(地址最高)，即基址指针；esp 始终指向栈顶(地址最低)，即栈指针。一般来讲，两者之间的区域叫作栈帧。

调用函数的函数称为“caller”，被调用的函数称为“callee”。
每调用一个函数，就会生成一个新的栈帧。在这个过程中：

  1. “调用者”需要知道在哪里获取“被调用者”返回的值；
  2. “被调用者”需要知道传入的参数在哪里，返回的地址在哪里。

在“被调用者”返回后，上述寄存器的值应该和调用前一致，因此需要使用栈来保存这些数据。

函数完成任务后会将 esp 移到 ebp 处，再弹出旧的 ebp 的值到 ebp。这样，ebp 就恢复到了调用前的状态了。

CMP 表示比较两个寄存器/内存中的值，结果会影响标志位寄存器。
标志位寄存器是一个 64 位的寄存器，其内部有很多标志位。
标志位是……先把这个寄存器看成 64 个二进制位，然后为帮助理解简化为 3 位，表示今天有没有吃早午晚饭。
什么都没吃就可以用 000，只吃了晚饭就可以用 001；这样就可以用这三位表示今天的状态，它们就是标志位。
标志位寄存器用来储存这些标志位，而 CMP 用来改变他们。
某种程度上，根据标志位来决定是否跳转。

比如，如果两个值相等，那么 ZF（用来标志两个值是否相等的零标志位）就会被置为 1，不相等就 0。
又比如，SUB ADD 等操作会产生溢出，和负数处理的情况都会影响标志位。
如果是正数减为负数， SF（符号标志位）就会被置为 1，如果是负数减为正数，SF 就 0，而 OF（溢出标志位）就会被置为 1，如果没有溢出，那么 OF 就 0，

CMP eax, ebx 等价于 SUB eax, ebx，即 eax - ebx，但不会将结果放回 eax，同时会影响标志位，进而影响跳转结果。

bss/data/text段

bss（Block Started by Symbol）段（segment）通常是指用来存放程序中未初始化的全局变量的一块内存区域，属于静态内存分配。
data 段通常是指用来存放程序中已初始化的全局变量的一块内存区域，属于静态内存分配。
code/text 段通常是指用来存放程序执行代码的一块内存区域。区域大小在程序运行前就已经确定，通常属于只读(某些架构也可写)。
其中也有可能包含一些只读的常数变量，例如字符串常量等。

一个程序本质上都是由 bss 段、data 段与 text 段组成的。
一般在初始化时 bss 段部分将会初始化/清零。
text 和 data 段都在可执行文件中，由系统从其中加载；而 bss 段不在可执行文件中，由系统初始化。

其他代码不变的情况下，将数组声明由 int ar[30000] 改为 int ar[300000] = {1,2,3,4,5,6}; 后，编译生成的文件体积会大幅增加。
两者的区别在于 data 段的大小：
函数内的自动变量都在栈上分配空间，.bss 不占用文件空间，.data 却需要。
bss 段并不给该段的数据分配空间，只是记录所需空间的大小；其大小从可执行文件中得到，然后链接器得到这个大小的内存块，紧跟在数据段后面。
data 段则分配，数据保存在目标文件中。它包含经过初始化的全局变量以及它们的值，当这个内存区进入程序的地址空间后全部清零。
包含 data 段和 bss 段的整个区段此时通常称为数据区。

下一步的 ret，将最顶端的母函数返回地址 pop 出来，esp 下移 4 个字节，同时 eip 回到母函数中的对应位置。
理所当然的，本题的修改应从返回地址处下手。

只要把母函数的返回地址篡改为 esp+4 ，eip 在执行 ret 后就会神奇的转移到栈空间中，指向返回地址的下方。
再把 shellcode 塞到 eip 的下方，就可以顺理成章地执行 sc 代码。
但是，为了让几十年后的 CTF 选手打起来更头疼，栈结构被设计的比想象中更复杂，总之不一定是 +4。
怎么有点喜欢这种一步一个小十八盘的题

跳过立即数，篡改的位置由指令的地址来确定。
一个比较基础的选择是 addr of “jmp esp”。esp 与 eip 此时形成了最佳搭配：
esp 移动到返回地址下方，shellcode 上方，eip 再把刚刚 pop 出来的地址读取，改为指向 esp 的位置。
此时，esp 与 eip 的位置完全相同。

从哪找这个指令呢？随便找一个系统里有的 dll，在里面随便找一行 jmp esp 都可以。
这些，就是大名鼎鼎的 esp 跳板。

（以上算是我第一次纯手打这么长的技术内容了）

本题前半部分与上题完全一致，依然第 76-80 字节为返回地址。这种情况还是可以直接 +4 的。
自然的，可从第 81 个字节开始（地址 0xbffffc90）存放 shellcode。
构建的字符串后赋”\x90\xfc\xff\xbf”，然后就……还没完，后面还要加 8 个”\xcc”。

终于问 AI 了：
CC 对应的汇编指令是 INT 3，“软件断点”。
下断点时，调试器把指令临时替换成一块 CC。当 CPU 执行到这，会触发一个中断信号（SIGTRAP），程序就会暂停运行，把控制权交给调试器。
这是为了确认程序停在了预期的地方，证明链路全通后，再把这段 CC 替换成真正的 shellcode。

直接运行程序，显示 Trace/breakpoint trap，我以前从来没想过还能人力下断点，计算机真奇妙！

躲得了初一躲不过未来十五，栈结构复杂性在这坑人：

程序启动时，操作系统会把环境变量和程序启动参数压入栈的最顶部。
在 GDB 内部运行时，GDB 会向环境中注入一些自己独有的环境变量，并且启动路径可能是绝对路径。
但在外部终端直接运行时，环境变量和 GDB 里不同，并且相对路径比绝对路径短。
这些变化导致栈的初始位置发生了错位，通常偏差在几十个字节左右。

所以在 GDB 里，0xbffffc90 指着 Shellcode 的第一个字节，但直接运行时，真实地址可能变成了 0xbffffca0。
当你还跳向 0xbffffc90 时，CPU 跳到了 Shellcode 前面的几分米处，或者跳到了 Shellcode 的一半，结果当然就是 Illegal instruction。

不用担心，还没到传说中的 ASLR。为您介绍 NOP Sled 滑板指令。
在返回地址与 shellcode 之间塞入大量的 NOP（\x90），数量随便，比如 114。
只要实际的返回地址落在了任意一个 \x90，CPU 就会一路执行 NOP，直到 Shellcode ，完成一次优秀的防抖。
在此做法中，返回地址要往高处不断调整，直到真正成功。本题我足足往上提了 64 个字节。
但是，这次什么都没有返回，原因是输入方式有误。

/bin/sh 发现文件结束了，于是光速跑路。

looks at what happens when you have restrictions on the return address.
This level can be done in a couple of ways, such as finding the duplicate of the payload ( objdump -s will help with this), or ret2libc , or even return orientated programming.
It is strongly suggested you experiment with multiple ways of getting your code to execute here.

研究了当对返回地址有限制时会发生什么情况。
可以通过几种方式实现这一级别，例如查找有效载荷的重复项（objdump -s 对此会有所帮助），或者使用 ret2libc，甚至使用面向返回的编程。
强烈建议您尝试多种方法让您的代码在此处执行。

代码长度超级加倍，目前我能一眼看出来的也就是返回地址的前两位不能为 BF。
（笑点解析：我一开始以为 BF 只有与 11 异或才等于 BF 其余都会生成其他值）
但就这一点已经足够灾难性了：所有栈地址的高位都是 BF，意味着 shellcode 不能再待在栈上。

fflush 强行刷新缓冲区，将其中暂存的数据立即写入到目标设备中。但是本题用不到他
v2 = (const void *)retaddr; 的实际代码为 ret = _builtin_return_address(0);，表示本次要操控这行代码上面的 gets 函数的返回地址。
(0) 返回当前函数的返回地址， (1) 返回当前函数的调用函数的返回地址， (2) 返回当前函数的调用函数的调用函数的返回地址。

新的地址从哪找？要么借助该程序其他代码，要么借助系统函数库。
一个主意是用 getpath 最后的 ret 的地址替换栈空间中 getpath 的返回地址，esp 下移后，eip 指向 ret，程序会再执行一次返回。
程序只检查第一次返回地址是否在栈上，第二次不管。然后就与上道题一样了，在栈上构建 shellcode。

如这张图，前面 80 个字节的缓冲区，ret 的地址即 0x080484f9 应放在 0xbffffc7c 处，紧接着的 fc80 放 fc84。
python -c 'print "A"*80+"\xf9\x84\x04\x08"+"\x84\xfc\xff\xbf"+"\xcc"*8'>exp.txt

但是，大部分程序都设置了 NX 位，eip 一旦运行到栈上会直接报错。
这时，需要返璞归半真，把返回地址设置为系统中可以执行 shellcode 的函数入口。
比如 system()，只要把路径作为参传进去，就可以帮你执行。
现在 getpath 的返回地址下方就是 system 的返回地址，这个地址夏季把填就行，关键在再下面的 system 参数，须设置为 /bin/sh。
这个字符串仍然需要在系统函数库中寻找，好在系统函数库足够庞大，找到这个不成问题。

system 的地址为 0xb7ecffb0。字符串路径通过下面的指令搜索：
strings -t d /lib/libc.so.6 | grep "/bin/sh"
得到相对 libc 起始点的地址为 0d01176511。libc 的起始地址通过 info proc map 查找显示为 0xb7e97000。
我笨，下载了个直观进行不同进制的运算与转换的命令行程序 qalculate-gtk。
qalc -b 16 0xb7e97000+1176511 = 0xB7FB63BF。

所以，python -c 'print "A"*80+"\xb0\xff\xec\xb7"+"A"*4+\xbf\x63\xfb\xb7"'>exp.txt。

introduces return to .text to gain code execution.
The metasploit tool “msfelfscan” can make searching for suitable instructions very easy, otherwise looking through objdump output will suffice.

引入了返回 .text 的方法来获取代码执行。
Metasploit 工具“msfelfscan”可以非常轻松地搜索合适的指令，否则查看 objdump 输出也足够了。

代码与上题唯一的不同在于返回地址的最高位只要是 B 就不行，直接把上道题的第二个方法排除了。
但第一个方法还是能用的，本题略。
那 esp 跳板呢？很遗憾，整个程序都找不到 jmp esp。
那就只能 ROP 了。

程序内有很多（也可能很少）短小的指令序列，他们是函数的一部分，以 pop 与 ret 为主，叫作 gadget。
快速寻找可以用 ROPgadget --binary，但受 protostar 性能限制，该工具和题目描述中的 msfelfscan 均未被搭载。
当他们在栈上串联起来后，每个 gadget 结尾的 ret 便可以当逗号使，所有的 gadget 构建成一个完整的武器。
理想的 payload 中没有指令，全是内存地址和数据。

本题作为入门题，只需要一个 gadget 就够了。
getpath 返回地址填目标 gadget 地址，栈空间下方给出 gadget 中需要 pop 的内容，最后再来一个返回到 NOP 的地址，后面就可以跟 shellcode 了
……因为寻找难度较大且没有 NX 机制，这道题还是把 shellcode 构建在了栈上。

也因此，本题的 gadget 地址是我从网上找的 0x08048492。
也由此，我学习到了肉眼找 gadget 的方法。

复现最难的一次，也让我长了一个教训：不能一昧的增加 NOP 的数量，优先选择把开始地址往高处提。
