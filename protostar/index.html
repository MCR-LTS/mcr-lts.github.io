栈是什么

一种后进先出的数据结构，满足调用函数的方式（调用时父前子后，返回时子先父后）。
栈支持 push 压入与 pop 弹出并存储到指定寄存器/内存中，生长方向从高地址到低地址。
pop 后，栈中的数据并没有被清除，只是无法直接访问。

栈帧是什么

一种专门用于保存函数调用过程中的参数、返回地址、本地变量等信息的栈。
ebp 指向栈底(地址最高)，即基址指针；esp 始终指向栈顶(地址最低)，即栈指针。一般来讲，两者之间的区域叫作栈帧。

调用函数的函数称为“caller”，被调用的函数称为“callee”。
每调用一个函数，就会生成一个新的栈帧。在这个过程中：

  1. “调用者”需要知道在哪里获取“被调用者”返回的值；
  2. “被调用者”需要知道传入的参数在哪里，返回的地址在哪里。

在“被调用者”返回后，上述寄存器的值应该和调用前一致，因此需要使用栈来保存这些数据。

CMP 表示比较两个寄存器/内存中的值，结果会影响标志位寄存器。
标志位寄存器是一个 64 位的寄存器，其内部有很多标志位。
标志位是……先把这个寄存器看成 64 个二进制位，然后为帮助理解简化为 3 位，表示今天有没有吃早午晚饭。
什么都没吃就可以用 000，只吃了晚饭就可以用 001；这样就可以用这三位表示今天的状态，它们就是标志位。
标志位寄存器用来储存这些标志位，而 CMP 用来改变他们。
某种程度上，根据标志位来决定是否跳转。

比如，如果两个值相等，那么 ZF（用来标志两个值是否相等的零标志位）就会被置为 1，不相等就 0。
又比如，SUB ADD 等操作会产生溢出，和负数处理的情况都会影响标志位。
如果是正数减为负数， SF（符号标志位）就会被置为 1，如果是负数减为正数，SF 就 0，而 OF（溢出标志位）就会被置为 1，如果没有溢出，那么 OF 就 0，

CMP eax, ebx 等价于 SUB eax, ebx，即 eax - ebx，但不会将结果放回 eax，同时会影响标志位，进而影响跳转结果。

bss/data/text段

bss（Block Started by Symbol）段（segment）通常是指用来存放程序中未初始化的全局变量的一块区域，属于静态内存分配。
data 段通常是指用来存放程序中已初始化的全局变量的内存区域，属于静态内存分配。
code/text 段通常指用来存放程序执行代码的内存区域。区域大小在程序运行前就已经确定，通常属于只读(某些架构也可写)。
其中也有可能包含一些只读的常数变量，例如字符串常量等。

一个程序本质上都是由 bss 段、data 段与 text 段组成的。
一般在初始化时 bss 段部分将会初始化/清零。
text 和 data 段都在可执行文件中，由系统从其中加载；而 bss 段不在可执行文件中，由系统初始化。

其他代码不变的情况下，将数组声明由 int ar[30000] 改为 int ar[300000] = {1,2,3,4,5,6}; 后，编译生成的文件体积会大幅增加。
两者的区别在于 data 段的大小：
函数内的自动变量都在栈上分配空间，.bss 不占用文件空间，.data 却需要。
bss 段并不给该段的数据分配空间，只是记录所需空间的大小；大小从可执行文件中得到，然后链接器得到这个大小的内存块，紧跟在数据段后面。
data 段则分配，数据保存在目标文件中。它包含经过初始化的全局变量以及它们的值，当这个内存区进入程序的地址空间后全部清零。
包含 data 段和 bss 段的整个区段此时通常称为数据区。

ROP 入门

（Return-oriented programming，返回导向编程）

栈溢出的控制点是 ret。
基础的栈溢出通过覆盖返回地址控制程序流，但只能使程序执行到某一个地址继续执行。
ROP 的主要目的便是利用程序中已有的小片段，在合适的位置布置一连串的返回地址，改变某些寄存器或者变量的值，从而控制程序的执行流程。

ROP 攻击一般需满足如下条件：

  • 程序存在溢出，并且可以控制返回地址。
  • 可以找到满足条件的 gadgets 及其地址。

从 32 位讲起。想要真正劫持程序需要 getshell —— system("/bin/sh")。
假设 system 的地址是 0xf7000000，/bin/sh 的地址是 0xdeadbeef，ROP 时，如何将后者传给前者？

一个参数的函数 foo，如果想要调用 foo(233)：

call foo 的一瞬间，foo 开始执行，RSP 指向返回地址，而它需要的参数位于 RSP + 4 的位置。
这意味着，如果通过 ROP 劫持返回地址到 foo，应确保进入函数的瞬间，RSP + 4 的位置是它的第一个参数，同时 RSP 为 foo 的返回地址。
将 foo 看做是 system，想要劫持，只需将栈覆写成下面这样：

地址 值 如果值为指针，其指向的地址 RSP
0x00000014 0xdeadbeef “/bin/sh”
0x00000010 0x00000000 空
0x0000000c 0xf7000000 system 的开始 <-
0x00000008 0x00000000
0x00000004 0x00000000
0x00000000 0x00000000

漏洞函数执行 ret 后会返回到 system，RSP 指向 0x00000010（system 的返回值）。system 要回到哪无所谓，get了shell 就行。
此时，RSP+4 为 "/bin/sh"，作为第一个参数，组装成了完整的 system，从而 getshell 成功。
把 system 的返回地址改成下一个函数的地址，可以实现按顺序调用多个函数。

再看 64 位，某种程度上比 32 位简单，只需控制寄存器的值就可以实现参数传递。
比如在旧版 gcc 中，程序都会被编译进一个函数用于初始化，其中一段汇编如下：

pop r15 占 2 字节，假设其起始地址为 0x8，ret 的地址就是 0xa。那么，0x9 这个地址合法吗？
是，而且很有用。如果从 0x9 开始看这个代码片段，由于地址错位，r15 会神奇地变成 rdi。
转变成的 pop rdi，让我们能够通过栈修改 rdi，实现参数传递。
这些有用的代码片段，一般称为 gadget。
gadget 通常指以 ret 指令结尾的代码片段，利用各种合适的 gadget 可拼凑出需要的程序逻辑。

覆盖栈如下：

地址 值 如果值为指针，其指向的地址 RSP
0x00000018 0x7f000000 system 的开始
0x00000010 0xdeadbeef “/bin/sh”
0x00000008 (gadget) pop rdi; ret <-

首先，原函数的 ret 返回到 pop rdi，RSP 指向 0x00000010；
pop rdi 将 0xdeadbeef 赋给 rdi，RSP 指向 0x00000018；
然后 gadget 中的 ret 返回到 system，RSP 指向 0x00000020，执行 "/bin/sh"，getshell 成功。

指令被执行时的 RSP，和它被触发时的不在同一个位置。
构造 Payload 时，放在 0x08 的地址是为了触发 gadget；而这个 gadget 真正需要操作的数据（比如传参），必须预先放在 0x10 等它，因为当 CPU 真正开始做动作时，RSP 刚刚滑落到那里。

32 位布局： system地址 + 系统返回地址 + "/bin/sh"地址
64 位布局： pop rdi; ret + "/bin/sh"地址 + system地址

ret2 basic

ret2 后面的内容即 ROP 链中各种函数、gadget 的来源。这并不是某个完整的利用方法，而是其中的一部分。

ret2text 就是利用虚拟地址空间的 ELF 部分中储存可执行代码的部分（即.text 段），利用点在原文件中寻找所需的代码即可。
ret2libc 则使用 dll 中的 gadget，相比 text 在利用时多出了一步，泄漏动态链接库的基地址。
这就要引入两个关于链接库装载的表：got （Global Offset Table，全局偏移表）和 plt（Procedure Linkage Table，过程链接表）。

因为 dll 加载时的基地址是随机的，所以当程序调用其中的函数时，实际是转跳到预留在程序中的函数入口，即 plt。
执行到 plt 后，会先检查对应的函数是否已经加载，如是则根据 got 表中记录的地址转跳实际的函数地址；否则先调用加载函数，再转跳到对应的函数。

加载后的函数地址储存在 got 表中，我们可以想办法输出表中的内容，减去偏移，从而实现泄漏 dll 的基地址，拿到后加上偏移就可以获得所需函数/gadget。
（dll 的加载是通过映射文件实现的，一个链接库映射到一段连续的内存空间）

何谓PLT与GOT

print_banner 函数内调用了 printf，而 printf 函数位于 glibc 动态库内，所以在运行之前，链接器无法知道进程运行起来之后 printf 函数的加载地址，真正运行后才能确定。

那么，进程运行起来之后，动态库也装载了，函数地址也确定了，call <printf的地址> 如何修改（重定位）呢？

printf 地址只能回写（运行时重定位）到数据段内，不能直接修改真正地址，因为：

  • 现代操作系统不允许修改代码段，只能修改数据段。
  • 如果 pb 函数在一个动态库（.so）内，修改了代码段，就无法做到系统内所有进程共享同一个动态库。

每个 .c 文件都会经历编译和链接两个阶段，其中编译阶段将 .c-源代码翻译成 .o-汇编指令的中间文件。
在 pb 函数的汇编中，call 的操作数是 0xfffffffc，即 -4，作为不知道的 printf 函数的地址的占位。
链接时，这里要根据 printf 地址（更确切的叫法应该是符号，链接器眼中只有符号，没有所谓的函数和变量）按照相对引用方式来修正。
这个过程称为链接时重定位，与运行时重定位的工作原理完全一样，只是修正时机不同。

链接阶段将中间文件（.o）通过链接器链接成一个可执行文件，主要完成以下事情：

  • 各个中间文之间的同名section合并
  • 对代码段，数据段以及各符号进行地址分配
  • 链接时重定位修正

除了重定位过程，其它动作无法修改 .o 中函数体内指令，而重定位也只能修改指令中的操作数，无法修改编译过程生成的汇编指令。

编译阶段，编译器无法知道 printf 函数在 glibc 运行库，而不是定义在其它 .o 中，只能生成调用 printf 的汇编指令。（链接阶段可以）
printf 在上述两种情况下都能工作。如果是后者，那在链接阶段，printf 地址已经确定，可以直接重定位，否则不能。

运行时重定位无法修改代码段，那在编译阶段就已生成好的 call 指令，怎么感知这个已重定位好的数据段内容呢？
链接器通过生成一段额外的小代码，获取 printf 地址并完成调用。

strcmp 与栈溢出

输入 11111111 显示错误，但输入 qqqqqqqq 显示正确。

strcmp 的返回值由 eax 接收，步过 strcpy 后，eax 的值为 1。
接下来 mov [rbp-4], eax 将返回值赋到 [rbp-4] 中，ret_num = 1。
buffer 长 8 个字节，因为输入的 qqqqqqqq 后面还有字符串截断符0×00，将 ret_num 由原来的 01 覆盖为 00，故最后输出成功。
但为什么 111111111 不行？原因在于 strcmp 的返回值差异以及小端序存储机制。

strcmp(str1, str2) 逐个字符比较 ASCII 码值，一旦遇到不同的字符，就返回两个字符的 ASCII 差值。
输入 “11111111”，第一个字符都是 1，但第二个字符 1<2，返回值为 49 - 50 = -1，ret_num = -1。
-1 在 32 位整型中的十六进制补码是 0xFFFFFFFF，0x00 溢出后，值变成了 0xFFFFFF00，依然是一个非零值，返回 true。

输入 “qqqqqqqq”，第一个字符 q>1，ret_num = 64 即 0x00000040，小端序排列是 40 00 00 00。
溢出后，最低位地址被覆盖，内存中的值变成了 00 00 00 00。ret_num = 0，返回 false。

总之，当输入的字符 ASCII 值小于正确密码（如 1 < 2），strcmp 返回负数，高位全为 FF，用 0x00 覆盖最低位无法将其变成 0。反之返回正数，高位全为 00。

堆由操作系统内核或者堆管理器动态分配，用于存放进程运行中被动态分配的内存段，只有在程序需要时才会分配，可动态扩张或缩减。
在 pwn 程序中，栈在程序加载进内存后就会出现，而堆由 malloc/alloc/realloc 函数分配内存后才会出现。

堆与栈相反，从低地址向高地址生长，可以申请到的内存空间比栈要大很多。
对堆的操作由堆管理器 (ptmalloc2) 实现，而非操作系统内核。因为程序每次申请或者释放堆时都需要系统调用，当频繁进行堆操作时，就会严重影响程序性能。

基本结构

使用中（分配后）：

chunk start：chunk的起始地址

pre_size：
如果该 chunk 物理相邻的前一地址 chunk（两个指针的地址差值为前一 chunk 大小）是空闲的话，那该字段记录的是前一个 chunk 的大小（包括头）（否则可记录数据）。
前面一个堆块在使用时并且 pre_size 为储存前面 chunk 的数据时, 他的值始终为0。

size：
用来指示当前堆块头部 (pre size+size) 加上 user data 的大小。必须是 2*SIZE_SZ 的整数倍，如果不是，会被转换为满足大小的最小整数倍。
32 位系统中，SIZE_SZ 是 4 ；64 位系统中，SIZE_SZ 是 8。
该字段的低三个比特位对 chunk 的大小没有影响, 字段的最后三位相当于三个 flag，作用分别是：
A (NON_MAIN_ARENA)：为 0 表示该 chunk 属于主分配区（主线程），为 1 表示属于非主分配区（非主线程）。
M (IS_MAPPED)：表示当前 chunk 从哪个内存区域获得的虚拟内存。为 1 表示从 mmap 映射区域分配，否则是从 heap 区域分配。
P (PREV_INUSE) ：记录前一个 chunk 块是否被分配。一般来说，堆中第一个被分配的内存块的 size 字段的 P 位都会被设置为 1，防止访问前面的非法内存。如果为 0，能通过 pre_size 字段获取上一个 chunk 的大小及地址，方便进行空闲 chunk 之间的合并。
前一个堆块的释放与否都和 pre_size、size->prev_inuse 的值有关, 便于内存的释放操作。

pre size (prev size) 和 size 头统称为 chunk 头。
每次 malloc 申请得到的内存指针，指向 user data 的起始处。

malloc (8) 申请到的堆块总大小为 16+8+8+1 = 0x21 (byte) ：
16 是系统最小分配的内存，8 是 pre size 字段的大小，8 为 size 字段的大小，以上如为 32 位数值减半。
1 是 P 位的值，只有 0/1 两个值。

空闲中（释放后）：

空闲中的 chunk 不存在 M 状态，只有 A|P 状态。
user data 头部被分配出 fd 和 bk，分别指向前后一个（非物理相邻）空闲 chunk 的起始地址，32 位占 4 字节, 64 位占 8 字节。
chunk 处于分配状态时，从 fd 字段开始是用户的数据。chunk 空闲时，会被添加到对应的空闲管理链表中进行统一管理。

释放后的 large block 中还存在另外两个成员，fd_nextsize 和 bk_nextsize，只有 chunk 空闲的时候才使用，用于较大的 chunk。
两者分别指向前后一个与当前 chunk 大小不同的第一个空闲块，不包含 bin 的头指针。
一般空闲的 large chunk 按照由大到小的顺序排列，在 fd 的遍历时，可以避免在寻找合适 chunk 时进行全体遍历。

malloc_chunk 结构

每个程序分配的内存在内部被”堆块”所替代，一个堆块是由元数据和程序返回的内存组成的（实际上内存是 malloc 的返回值）。
所有的堆块都保存在堆上，这块内存区域在申请新的内存时会不断扩大，当一定的内存释放时也可以收缩。

堆块大小与空间复用

64 位操作系统中，用户分配到的最小堆块大小为 33B：
prev_size (8) + size (8) + fd (8) + bk (8) + next_chunk -> p (1B)
若用户申请的大小超过最小堆块大小，会与 16B 进行对齐。
32 位操作系统数值减半，除 p 外。

当一个 chunk 处于使用状态时, 它的下一个 chunk 的 prev_size 无效。所以下一个 chunk 的 prev_size 也可以被当前 chunk 使用，这就是 chunk 的空间复用。

申请堆块的本质

堆管理器 ptmalloc2 主要通过 malloc/free 函数来分配和释放内存块。
通俗地讲，它就是一个”中间商”，在程序想要申请向系统申请堆空间时，p2 就会申请一块很大的空间，并根据算法从这些内存中把空间真正的分配给程序。
