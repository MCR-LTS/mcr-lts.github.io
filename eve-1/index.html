帮你试过了，Apple Silicon Mac 无法安装 EVE-NG

帮你试过了，Apple Silicon Mac 无法安装 EVE-NG

Mercurylotus03/12/26

封面：EVE - smile (album)

引引言

本文原来的标题是《M3 Mac EVE-NG 之一》。
一开始，我抱着一股一定能成，把不可能变成可能的信念；然而，经过两天的纠缠，转译问题始终无法解决，遂放弃。
本文依旧留作纪念与警醒。其中与 EVE 直接相关的配置环节是通用的，可以直接拿来参考。

引言

我的主线任务已注定是网络安全，不过我身边的一位同学，虽然不是网络专业的，但对计算机网络与路由配置的理解已秒杀我校所有网络工程与网络安全的学生。
看到他行云流水的配置 ip，操作 Proxifier，并在 pve 和 PC 间运转自如，倍感羡慕，深受鼓舞，值此学校课程开设之际，记录一下安装配置使用的过程与体会。
EVE 难不难运行并不重要，重点在于，我想学会配套工具的使用，并理解路由的配置，跟随大佬的脚步。

下载

EVE 服务端目前没有支持 ARM 架构的计划，主流虚拟机软件也没法转译 x86 的 iso，所以试试可以模拟运行 x86 的 UTM。(4.7.5)
再从 EVE 官网下载 Community Edition，痛失 3.2GB VPN 流量。

打开 UTM，依次新建，模拟，其他。内存推荐 8192M，核心数选 4，旧版硬件不勾。
下一页，上传 iso 文件后，取消勾选 UEFI 启动。
存储空间……勉强给到 40GB，官方的最低要求。
共享目录路径，那个不需要。
最后给虚拟机起一个你喜欢的名字，配置就完成了。

安装

启动虚拟机，然后泡壶茶坐着等。等喝完一杯以后，会出现选择语言的页面，其实你别无可选只有英语。接下来的键盘布局倒是可以选择中文。最后确认安装，然后睡个午觉。

然后第一次重复上述过程…第二次…
我知道无限重复并非故障，而是分卷安装的正常一环，每个分卷都要重复一遍安装。
然而就在这循环中，我开始思考…

全部木大

恍然意识到，以上又与我的成长方向背道而驰了。
我要锻炼 AI 的准确性以及写 prompt 的能力，就应该把这个全盘交给 AI 完成的。

老师明明发布了文件，我却非要自己找；
明明有知识的高效集合，我却非要查阅教程…
时代车轮滚滚前行，在效率为王的时代，花时间感受路上的风景变得越来越亏。
而为了跟上时代，我别无选择，直觉亦不容犹豫。

我们推倒重来，除了保留 UTM 外。

下载（1）

请给出一份 Apple Silicon Mac 使用 UTM+老师提供的 ovf 压缩包安装 EVE-NG 服务端的详细操作指南，无需分析可行性与推荐其他方案。给你一份我们老师发的普通 windows 设备使用 workstation 安装的操作指南作为参考。
老师提供的文件 (pwd:93z0) 列表：
模拟器相关文件.zip（下载以前不知道里面是什么）
……
此外，官网的 Apple silicon 原生客户端包包括：
……
生成教程时，可以辅助搭配我在本机上装的 arm windows 虚拟机使用，但我要挑战自己，能在 mac 上完成的就不换虚拟机。
先从老师发的文件，我应该下载哪些开始。

在 Mac 上，我只需要下载老师提供文件中的两个 zip。

依旧打开 UTM，新建，模拟，其他，CPU 核心数选 4（是个偶数就行），再下一步启动设备选“无”并取消勾选 UEFI 启动。

驱动器大小随便填，因为“下一步马上会删掉这个默认硬盘”。跳过共享目录，保存。

在虚拟机编辑页面中，删除唯一的驱动器，再点击新建。

解压 EVE-COM-5.0.1-12.zip，看到个贼 TM 大的 .vmdk，在刚才的页面点击导入，选择这个文件，其余均不变，直接保存。

安装（1）

启动虚拟机，然后泡壶茶坐着等。此时 CPU 温度会飙升至 80 左右，可以用散热管里的水直接泡茶。不同于 iso 文件，现在使用的 ovf 文件开箱即用，可以直接输入用户名 root 与密码 eve 。

接下来一路回车就行了，直到重启。

在浏览器输入服务端上方显示的 http://192.168.64.3/，非常顺利的打开了登录页面！然后输入用户名密码选择 html 终端……最后登不进去。用 df -h 看了一下，存储空间也没满。

排查半小时后，发现客户端和服务端的账号不一样，前者是 admin。我 nm

终于进入了 console 页面。在第一步运行完成以后，电脑的温度就降到正常水平了，看来转译运行也没那么恐怖。我有过转译 x86 docker 运行 pwn 虚拟机的前车之鉴，还以为温度又得起飞来着（

不是我他妈真没绷住……AI 输出的两组帐密都是正确的，我直接当成幻觉了，难怪我 CTF 做不好……

上传镜像

仅从名字来看，Mac 就不支持 WinSCP。我同学使用 Bitvise，但是它也只有 Windows 版……于是，我准备先安装 Mac 原生可用的 FileZilla (3.69.6)，然后再用 wine (10.0 in crossover 25.1) 转译运行 Bitvise (9.59)。

强烈安利 macked.app ，我好几个软件的破解版都是从这下的，几乎全免费且均提供直链下载。

FileZilla

虽说破解版好，但由于这个应用与安全息息相关，破解破坏了一些检查机制，使应用无法正常运行。
鉴于正版能够使用全部核心功能，还是老实用正版吧。

打开应用，授予用户目录访问权限后就可以进入主界面。

我们可以看到，新建连接的 GUI 直接放在了顶上。四个空分别填写192.168.64.3 root eve 22 ，press the button。过程中所有权限全部始终授权，便可成功连接，上图中的右侧将会显示远程站点的目录。

使用该应用来上传核心镜像文件。EVE 对镜像文件夹的命名有严苛要求，必须是 模板名-版本号 的格式，全小写或完全匹配 yml 前缀。

在右侧进入 /opt/unetlab/addons/qemu/，创建名为 Ruijieroute-1.0 与 Ruijieswitch-1.0 的两个目录，依次进入，把老师发的对应 hda.qcow2 文件拖进去。

在 eve 服务端输入 /opt/unetlab/wrappers/unl_wrapper -a fixpermissions ，修复权限。

此时应该把一些 py，yml 都放进去，但我想测试一下第二个应用，把这部分轻活交给它干。

Bitvise

如图所示配置。

点击左侧 New SFTP window ，又看到了与 fz 类似的样式。

Ruijieroute.yml 和 Ruijieswitch.yml 放到 /opt/unetlab/html/templates/intel/ 与 /opt/unetlab/html/templates/amd/，ruijieroute.png 和 ruijieswitch.png 放到 /opt/unetlab/html/images/icons/，config_ruijie.py 放到 /opt/unetlab/scripts/，最后再执行一遍修复权限，断开连接。

至此，搭建环节已全部完成。接下来开始操作 EVE-NG 的 GUI。

客户端提供两种 console，HTML5 可以直接在浏览器里操作，省时省力；然而感兴趣的同学亦可尝试 Native console —— 本地配置模式，需要调用 CRT 进行设备配置，但每次都要安装插件才能使用。

这里也不做过多赘述，Mac 端的替代品是公认最全能的 iTerm，根本不需要再安装别的。如果选择 Native，未来如需使用 console，系统会自动打开 iTerm 运行。

还是忍不住炫耀一下我的 iTerm，这套主题两年过去经久耐看（

创建

登录后在主页面选择 Add new lab，随便起个名，别的可以不改，会进入密集恐惧症的福音——只有网格的实验页面。退出重登以后，系统会默认打开此页面，除非你主动在网格页面左侧选择 Close lab。如果还想再回来，在主页面点击这个实验，右侧选择 Open 即可。

要将节点（设备）添加到拓扑图中，在左侧选择 Add object > Node，或者右键网格空白处选择 Node。如果在 Node 列表结尾处看到了添加进去的锐捷模拟器并且能正常使用，说明上篇文章的搭建取得了最终胜利。

除了两个锐捷，还可以看到 Virtual PC，它用来进行测试验证。在这里先添加一个 Ruijie vRoute ，配置一点也不用改直接 Save。为什么 CPU 核心数不能再分到小数点后两位啊

Number of nodes to add：字面意思

Name/Prefix：节点名字。如果添加多个节点，名字将作为前缀，其后附加数字编号。

CPU/RAM：预设值，根据自身情况进行调整，建议分别设置成 1 和 1000MB。

Console：控制台登录方式，默认 telnet，联动 CRT（也可选 VNC/RDP）。

生成一个灰色圆盘并且鼠标放上去后出现一个橙色的痔插头说明没问题，按照相同的步骤和默认配置再添加一个 Ruijie vSwitch 和 Virtual PC(VPCS)，接下来就是痔趣相投连接插头的环节了。

连接

[!WARNING]

设备启动后，无法修改接线，必须在启动前完成修改。

找到一个节点，拿起他的插头（鼠标左键长按直到）拖到第二个节点上，松开鼠标会出现连接窗口，选择你想连接的接口后保存。首先连接路由器和交换机，再连接交换机和主机，接口依旧选择默认。

说实话这些全是默认我都怀疑是老师提前给我们配置好了，因此或许这些步骤没有参考意义，试试才知道

鼠标右键节点，Start 开机，节点由灰色变为蓝色（或在节点名称左侧出现 ▸ 图标），启动成功。别的指令都是字面意思，而下面两个

Wipe node：擦除节点，该功能将清除节点的NVRAM。每次更改节点设置（如：CPU、RAM、启动配置等）都必须在节点上进行该操作。

Capture：抓包功能，可以联动Wireshark进行抓包。

此时的拓扑图应如下所示。

然而我的图标变蓝后，过了 5 秒钟又变回灰色，十分诡异。转译产生的底层进程崩溃问题还在追我。
不过多打开几次

内存分到 8192，但是核心数反而要降低成 2。

越多的 CPU 核心 = 越慢的灾难

在原生系统里，核心越多越快。但在 “M芯片纯软件翻译 x86 代码 (TCG)” 的场景下，情况截然相反：
x86 架构有非常严格的“内存指令顺序”，而 ARM 架构的内存顺序是乱序的。为了在这两者间安全翻译，UTM 必须插入海量的“内存锁”。 当你给了它 8 个核心时，这 8 个核心在翻译底层代码时，每毫秒都在疯狂争抢“线程锁”，导致 Mac 的算力全耗在了“内耗排队”上，反而让原本就很慢的开机速度雪上加霜！

随后右键 Wipe 一遍，就可以顺利启动。但使用起来是否顺利，有待后续观察。

如要删除设备，右键节点先停用（如果已经启用的话），变为灰色后才可以删除。
