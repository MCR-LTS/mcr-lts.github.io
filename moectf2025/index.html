What is upx UPX（Ultimate Packer for eXecutables）是一款著名的开源、高性能可执行程序文件压缩工具，主要用于将Windows、Linux、macOS上的PE、ELF等二进制文件（.exe, .dll, .so等）体积缩小 50%-70%，且压缩后的文件无需解压即可直接运行。
核心功能与特点：
- 体积减小： 显著降低软件分发时的安装包大小和磁盘空间占用。
- 即时解压： UPX压缩文件内置了自解压代码，程序运行时会自动在内存中解压并执行，对大多数应用无明显性能损耗。
- 支持多格式： 支持Windows程序/DLL、Linux可执行文件和内核、macOS应用等。
- 加壳工具： UPX常作为一种“壳”工具，可用于对程序进行保护，防止被轻易修改和破解。
- 安全与自由： 完全免费、开源，使用UCL或LZMA算法，在现代机器上具有极快的解压速度。
用途与争论：
UPX被广泛用于开发人员在分发程序时减小体积。但由于其压缩和加壳特性，也曾被木马和病毒用于隐藏代码，躲避杀毒软件的查杀。

Buffer[i] = 预设值[i] ^ Buffer[i+1] ^ 0x21

[!NOTE]

哪些是“垃圾信息”？（解题时可以直接无视）

在读 C++ 伪代码时，以下特征的代码可以直接在脑内打马赛克，它们几乎不可能对解题有影响：

1. 栈保护机制

  • 这是什么： 防止栈溢出的系统底层机制。
  • 为什么无视： 只有打 Pwn 题才会关心这个。在纯逆向分析算法时，它毫无意义。

2. 局部变量的析构函数

  • 这是什么： C++ 是面向对象的，局部变量（如字符串）用完后必须释放内存（析构）。
  • 为什么无视： 它是编译器自动加上去“擦屁股”的，不包含任何出题人的加密逻辑。

3. 内存分配器

  • 这是什么： C++ std::string 在底层申请堆内存时需要的组件。
  • 为什么无视： 同上，纯系统底层实现。

4. 繁琐的标记变量

  • 这是什么： 反编译器在处理复杂的 C++ if/else、异常处理或提前 return 时，往往会生搬硬套出很多 bool 型的临时变量（如这里的 v3, v4）。
  • 为什么无视： 你只需要看懂它最终导致了什么结果（比如 v4=0 就代表不进入 Format Error 的报错），不需要去琢磨作者是不是真的写了 v3=0 这种代码（作者大概率没写，只是编译器优化导致反汇编成了这样）。

什么是“Z3 数据”？为什么它不是普通数字？

在普通的 Python 编程中，数字就是数字。当你写下 x = 5 时，内存里确实有一块空间存着二进制的 0000 0101。 如果你写 y = x + 2，Python 会立刻算出 7，把 7 存到 y 里面。这叫具体执行。

Z3 工作的原理完全不同。当你写下 x = BitVec('x', 32) 时，内存里根本没有数字，Z3 给你创建的是一个“代数符号”（一个空盒子，贴着标签叫 x）。

接着写下 y = x * 2 + 5 时，Z3 不会去做任何计算（因为 x 里面根本没有数字）。Z3 会把这个公式记在一个“笔记本”上。此时的 y，在 Python 眼里是一个“Z3 的数学表达式对象”。

它在内存里的长相其实是棵树（AST 抽象语法树）：

这就是不能直接对 Z3 中算出来的结果用 chr() 的原因——因为 chr() 需要一个数字，但你传给它的是一棵名为 ((x \* 2) + 5) 的“数学语法树”对象！

破案后的提取：.as_long() 是跨界传送门

当 solver.check() == sat 时，代码告诉我们：“我根据你这堆公式算出来了， x 如果是 65 就能满足所有条件。”

此时，答案被存放在了 model 里。 写下 model[flag[0]] 时，取出来的依然是一个 Z3 的特定对象（ IntNumRef 或 BitVecNumRef），它代表“在 Z3 里，这个符号的值确定为 65”。

我们要把它拿来使用，必须调用它自带的方法 .as_long()，其作用是“把 Z3 里的特殊数字对象，转换成 Python 里的 int。” 之后，chr(65) 才能正常工作，把它变成大写字母 'A'

看到什么特征，一定要掏出 Z3？

如果只是一些简单的异或 ^ 或是加减法，直接自己写逆向求反就可以，但如果在伪代码里看到了以下特征，就要选择 Z3：

特征 1：变量互相死死纠缠

  • 代码长相：b[i] = b[i+1] ^ b[i-1] + a[i]。
  • 为什么用 Z3：要想知道现在的字符，得先知道上一个字符；可要知道上一个字符，还得知道下一个字符。Z3 只解全局方程组，不讲究先后顺序，它最喜欢这种一锅乱炖的逻辑。

特征 2：数学运算与“位运算”疯狂混用

  • 代码长相：result = ((input * 342) ^ 0xDEADBEEF) + (input >> 4) & 0xFF
  • 为什么用 Z3：
    • 如果是纯数学公式（比如矩阵乘法），可以用线性代数工具解；如果是纯位运算（比如多次异或），可以用密码学特性自己推倒。
    • 但如果加法乘法（会产生进位）和异或移位（破坏进位）混在一起，在纯数学上它是极其难逆向的（也就是所谓的非线性变换）。
    • 而 Z3 的 BitVec（位向量）是专为模拟算术逻辑单元设计的，对于这种混合碾压，它瞬间搞定。

特征 3：多元超长方程组

  • 代码长相：

  • 为什么用 Z3：人类的大脑解三元一次方程组已经是极限了，写脚本爆破多变量的循环会直接让 CPU 冒烟。Z3 底层采用专门算法，解 50 元方程组只需 0.1 秒。如果你看到一长串极其繁杂的 if 将你的几个输入字符加减乘除混在一起比较，直接全部丢给 Z3。

A cup of tea

可以了解一下TEA加密

TEA（Tiny Encryption Algorithm，微型加密算法） 是一种以极度简单和高效著称的对称密钥分组密码。

TEA 算法的每轮加密只使用了极其基础的算术和逻辑运算：加法、异或、位移。

为了防止每一轮的加密过程完全相同（对称性攻击），它引入了一个常数 **Delta (0x9E3779B9)**。 这个数字来源于黄金分割率（$\frac{\sqrt{5}-1}{2} \times 2^{32}$）。在每一轮计算中，Delta 会不断累加，提供不同的非线性扰动。

TEA 的加密核心通常只有短短几行代码。

解密代码与加密代码几乎完全对称，只是将 sum 初始化为 delta * 32（即 0xC6EF3720），然后进行减法运算即可。

函数原型：WinMain 参数

这是图形程序的入口点（相当于控制台程序的 main）：

  • hInstance: 当前程序的实例句柄 (可以理解为程序加载到内存后的基址标志)。后面加载图标、资源都要用到它。
  • lpCmdLine: 命令行参数。
  • nShowCmd: 窗口的初始显示模式（比如默认展示、最大化展示、隐藏等）。

第 1 阶段：加载全局资源

  • 作用：从程序的 .rsrc 资源段中提取字符串。0x67 和 0x6D 是资源 ID。
  • 解读：这就是在字符串里搜不到文本的原因。程序把“窗口标题”和“窗口类名”存在了本地化资源表中。

第 2 阶段：注册窗口类

  • 解读：在画出窗口之前，必须先向系统提交一份“图纸”（定义长什么样、用什么图标、什么鼠标、谁来处理逻辑）。
  • 这里只有 lpfnWndProc 是有用的，其他像图标、背景全是为了让窗口能正常跑起来的废话，一律跳过。

第 3 阶段：实例化窗口

  • 作用：向系统申请内存，真正把界面上的窗口创建出来（但这时窗口还没显示出来）。

第 4 阶段：展示窗体：让用户能看到界面。（代码略）

如果看到以下三个特征同时出现，大概率就是RC4：

  • 常数 256（十六进制通常显示为 0x100 或 255 / 0xFF 的位运算）。
  • 数组初始化：一个长度为256的数组被赋值为 0, 1, 2 ... 255。
  • 元素交换操作：频繁出现形如 temp = a; a = b; b = temp; 的交换逻辑。

在反编译代码中，RC4 必定分为两个核心函数（或者在同一个函数里的两大块逻辑）：

第一阶段：密钥调度算法（KSA）—— 经典的两层循环。作用是根据传入的密钥（Key）来打乱一个 256 字节的 S 盒。

第二阶段：伪随机流生成算法（PRGA）—— 加密/解密过程。作用是生成密钥流，并与明文进行 异或（^） 操作。

前面的特征没看到，ida 的反汇编能力可没这么强。后面看的倒是挺清楚。

解题时，找齐两个东西：

  • Key（密钥）：看传入的字符串是什么，可能是硬编码在程序里的明文，也可能是动态生成的一段数据。
  • Ciphertext（密文）：看最后被异或的数组，通常是全局变量里的一串乱码数组（hex值）。

紧挨着 rc4 函数的还有个 hex_to_binary ，逻辑相信是不用深究的。重点是，我还一个密钥密文都没找到……

双击主函数中的 enc 与 key 数组，分别得到 _Z3keyB5cxx11 _Z3encB5cxx11 和一堆问号。这就是“全局对象的动态初始化”。

在汇编和编译原理中：

  • .data 段：存放已初始化的全局变量（比如 C 语言里的 char key[] = "123";）。
  • .bss 段：存放未初始化或初始化为0的全局变量。这些空间在程序静态存放在硬盘上时是不占硬盘体积的，只有程序运行加载进内存时，操作系统才会给它们分配空间并清零，所以你在静态分析（IDA不运行程序）时，看到的只能是 ?。

那为什么 Key 和 Enc 会在 .bss 段？ 因为它们是 C++ 的 std::string 类型，作为全局变量时，必须要执行构造函数才能完成赋值。这个过程发生在程序启动后、进入 main 函数之前。 所以，这些字符串的值并不是一开始就写死在 .bss 里的，而是在程序刚刚跑起来的一瞬间，由隐藏的启动代码“填入”进去的。
