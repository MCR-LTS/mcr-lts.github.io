What is upx

UPX（Ultimate Packer for eXecutables）是一款著名的开源、高性能可执行程序文件压缩工具，主要用于将 PE、ELF 等二进制文件（.exe, .dll, .so 等）体积缩小 50%-70%，且压缩后的文件无需解压即可直接运行。

核心功能与特点：

  • 体积减小： 显著降低软件分发时的安装包大小和磁盘空间占用。
  • 即时解压： UPX压缩文件内置了自解压代码，程序运行时会自动在内存中解压并执行，对大多数应用无明显性能损耗。
  • 支持多格式： 支持Windows程序/DLL、Linux可执行文件和内核、macOS应用等。
  • 加壳工具： UPX常作为一种“壳”工具，可用于对程序进行保护，防止被轻易修改和破解。
  • 安全与自由：使用UCL或LZMA算法，在现代机器上具有极快的解压速度。

UPX被广泛用于在分发程序时减小体积，但其压缩和加壳特性也曾被木马和病毒用于隐藏代码，躲避杀毒软件的查杀。

加密公式是 v14[i] = Buffer[i] ^ Buffer[i+1] ^ 0x21 （当不是最后一个字符时）
程序使用 fgets 读取输入，会把敲击的回车符 \n (0x0A) 一并读入，因此如果输入的 flag 有 35 个字符，v6（字符串长度）实际上是 36。 这意味着计算第 35 个字符（下标 34）时，34 < 36 - 1 的条件依然成立。

前向公式：预设值[i] = Buffer[i] ^ Buffer[i+1] ^ 0x21
根据异或运算的自反性，后向推导公式就是：
Buffer[i] = 预设值[i] ^ Buffer[i+1] ^ 0x21

已知 Buffer[35] = 0x0A，从后往前逐个计算出原本的字符：

  • Buffer[34] = 0x56 ^ 0x0A ^ 0x21 = 0x7D （即}）

  • Buffer[33] = 0x24 ^ 0x7D ^ 0x21 = 0x78 （即x）

  • Buffer[32] = 0x29 ^ 0x78 ^ 0x21 = 0x70 （即p）

以此类推。

异或运算特性：如果 A ⊕ B = C，那么 A = C ⊕ B。

[!NOTE]

哪些是“垃圾信息”？

在读 C++ 伪代码时，以下特征的代码几乎不可能影响解题：

  1. 栈保护机制：防止栈溢出的系统底层机制

只有打 Pwn 题才关心这个，纯逆向分析算法时，它毫无意义。

  1. 局部变量的析构函数

C++ 是面向对象的，局部变量（如字符串）用完后必须释放内存（析构）。它是编译器自动加上去“擦屁股”的，不包含任何加密逻辑。

  1. 内存分配器：std::string 在底层申请堆内存时需要的组件

同上，纯系统底层实现。

  1. 繁琐的标记变量

IDA 在处理复杂的 if/else、异常处理或提前 return 时，往往会造出很多 bool 型的临时变量（如这里的 v3, v4）。

只需要看懂它最终导致了什么结果（比如 v4=0 代表不进入报错），不需要去琢磨是不是真的有这种代码，往往只是编译器优化导致反汇编成了这样。

什么是“Z3 数据”？为什么它不是普通数字？

在普通的 Python 编程中，数字就是数字。写下 x = 5 时，内存里确实有一块空间存着二进制的 0000 0101。 如果你写 y = x + 2，Python 会立刻算出 7 并存到 y 里面，这叫具体执行。

Z3 工作的原理完全不同。当写下 x = BitVec('x', 32) 时，内存里根本没有数字，Z3 创建的是一个“代数符号”（一个空盒子，贴着标签 x）。

接着写下 y = x * 2 + 5 时，Z3 不会做任何计算（因为 x 里面没有数字），但会把它记在一个“笔记本”上。此时的 y 在 Python 眼里是一个“Z3 的数学表达式对象”。

它在内存里的长相是棵树，AST 抽象语法树：

这就是不能直接对 Z3 算出来的结果用 chr() 的原因——它需要一个数字，但传给它的是一棵名为 ((x \* 2) + 5) 的“数学语法树”对象。

当 solver.check() == sat 时，代码告诉我们：“我算出来了， x 如果是 65 就能满足所有条件。”

此时答案被存放在了 model 里。写下 model[flag[0]] 时，取出来的依然是一个 Z3 的特定对象（ IntNumRef 或 BitVecNumRef），它代表“在 Z3 里，这个符号的值确定为 65”。

要把它拿来使用，必须调用它自带的方法 .as_long()，把 Z3 里的特殊数字对象转换成 Python 里的 int。 之后，chr(65) 才能正常工作，把它变成大写字母 'A'

看到什么特征，一定要掏出 Z3？

如果只是一些简单的异或/加减法，自己写逆向求反就可以；但如果在伪代码里看到了以下特征，就要选择 Z3：

特征 1：变量互相死死纠缠

要想知道现在的字符，得先知道上一个字符；可要知道上一个字符，还得知道下一个字符。Z3 只解全局方程组，不讲究先后顺序，最适合这种一锅乱炖的逻辑。

特征 2：数学运算与“位运算”疯狂混用

如果是纯数学公式（比如矩阵乘法），可以用线性代数工具解；如果是纯位运算（比如多次异或），可以用密码学特性自己推导。

但如果加乘法（会产生进位）和异或移位（破坏进位）混在一起，在纯数学上是极其难逆向的（也就是非线性变换）。

Z3 的 BitVec（位向量）是专为模拟算术逻辑单元设计的，对于这种混合碾压，它瞬间搞定。

特征 3：多元超长方程组

Z3 底层采用专门算法，解 50 元方程组只需 0.1 秒。如果你看到一长串极其繁杂的 if 将你的几个输入字符加减乘除混在一起比较，直接全部丢给 Z3。

特征 4：复杂的迷宫路线 / 数独题 / 状态机题

比如一个巨大的 while(1) 循环里面有一堆 switch/case，数组里有几个指针 x, y 走迷宫：

TEA（Tiny Encryption Algorithm，微型加密算法）是一种以极度简单和高效著称的对称密钥分组密码。

TEA 算法的每轮加密只使用了极其基础的算术和逻辑运算：加法、异或、位移。

为了防止每一轮的加密过程完全相同（对称性攻击），它引入了一个常数 Delta (0x9E3779B9，来源于黄金分割率)。每轮计算中，Delta 会不断累加，提供不同的非线性扰动。

TEA 的加密核心通常只有几行代码。

解密代码与加密代码几乎完全对称，将 sum 初始化为 delta * 32，然后进行减法运算即可。

从正常流（try）到异常流（catch）之间，没有任何显式的跳转指令。 如果 IDA 没有找到明显的异常动作，就会认为“solve() 永远不会抛出异常，catch 块的代码永远不可能被执行”，并直接从伪代码里抹除。

如果看到以下三个特征同时出现，大概率就是 RC4：

  • 常数 256（十六进制通常显示为 0x100 或 255 / 0xFF 的位运算）。
  • 数组初始化：一个长度为 256 的数组被赋值为 0, 1, 2 ... 255。
  • 元素交换操作：频繁出现形如 temp = a; a = b; b = temp; 的交换逻辑。

在反编译代码中，RC4 分为两个核心函数（或者在同一个函数里的两大块逻辑）：
第一阶段：密钥调度算法（KSA）—— 经典的两层循环。作用是根据传入的密钥打乱一个 256 字节的 S 盒。
第二阶段：伪随机流生成算法（PRGA）—— 加密/解密过程。作用是生成密钥流，并与明文进行异或操作。

前面的特征没看到，IDA 的反汇编能力没这么强。还好后面的看到了。

  • Key：看传入的字符串是什么，可能是硬编码在程序里的明文，也可能是动态生成的一段数据。
  • Ciphertext：看最后被异或的数组，通常是全局变量里的一串乱码数组（hex值）。

在汇编和编译原理中：

  • .data 段：存放已初始化的全局变量（比如 char key[] = "123";）。
  • .bss 段：存放未初始化或初始化为0的全局变量。这些空间在程序静态存放时不占体积，只有程序运行加载进内存后，操作系统才会给它们分配空间并清零，所以静态分析时看到的只能是 ?。
